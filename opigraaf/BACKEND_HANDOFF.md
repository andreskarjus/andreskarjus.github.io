## Current Client-Side Edit Feature

Edit mode is present in:

- `opigraaf.html`

It is generated by:

- `build_v2_graph_viewer.py`

Edit mode is off by default. When enabled, the selected-node panel exposes editing controls.

Supported client-side edits:

- Rename node
- Delete node from editable view
- Add link from selected node to another node
- Remove existing or added links

Edit persistence today:

- The embedded original graph is immutable.
- Edits are stored as a patch overlay.
- Every edit creates a history state.
- Undo, redo, jump-to-history-state, revert-to-original, and clear-saved-edits are implemented.
- History is stored in browser `localStorage` under:
  - `oppekava_v2_curriculum_focus_edit_history_v1`
- Export downloads a JSON payload containing current patch and full history.

Patch shape:

```json
{
  "renamed": {
    "node_id": "New label"
  },
  "deletedNodes": ["node_id"],
  "addedEdges": [
    {
      "id": "edit_edge:...",
      "source": "node_id",
      "target": "node_id",
      "type": "related_to",
      "confidence": 1,
      "authority": "expert_edit",
      "source_url": null
    }
  ],
  "deletedEdges": ["edge_id"]
}
```

Important implementation detail:

- Browser automation has trouble directly reading page globals in this environment, but the viewer itself exposes a debug/test handle:
  - `window.graphEditor`
- The UI should be manually tested in a normal browser after future changes, especially edit controls and export.

## Recommended Backend Implementation Plan

### 1. Decide The Persistence Contract

Keep the client patch format as the canonical first version. Do not try to mutate `nodes.jsonl` and `edges.jsonl` directly from the browser.

Recommended saved payload:

```json
{
  "snapshot": "v2_curriculum_focus",
  "expert_name": "Name",
  "history_index": 4,
  "current_patch": {},
  "history": [],
  "client_version": "v1"
}
```

Store complete snapshots periodically, plus append individual edit events for auditability.

### 2. Add Session Controls To The Viewer

Add a small backend panel inside edit mode:

- Backend URL input, default `http://127.0.0.1:8790`
- Expert name input
- Create/load session
- Save now
- Auto-save toggle
- Last saved timestamp/status

Keep the static-only workflow intact. If backend calls fail, the viewer should continue using localStorage and export.

### 3. Wire Save/Load API Calls

Client actions:

- `POST /sessions` to create a session
- `POST /sessions/{id}/events` after each edit, or batch after debounce
- `POST /sessions/{id}/snapshots` after each history commit or every N seconds
- `GET /sessions/{id}` to load prior saved state

Implementation note:

- On load, set `state.history` and `state.historyIndex` from the latest saved snapshot, then call:
  - `updateHistoryUI()`
  - `renderSelection()`
  - `updateVisible()`
  - `fit()`

### 4. Harden Patch Validation Server-Side

Add validation in `edit_backend.py`:

- Confirm renamed node IDs exist.
- Confirm deleted node IDs exist.
- Confirm deleted edge IDs exist or are added edit-edge IDs.
- Confirm added edge endpoints exist and are not deleted.
- Confirm edge type is from a known set or explicitly mark as expert-created.
- Reject huge payloads.

The graph snapshot path can be configurable, but default to:

- `curriculum_graph_et\graph\snapshots\v2_curriculum_focus`

### 5. Add Export/Import Utilities

Add scripts:

- `scripts/export_edit_session.py`
- `scripts/apply_edit_patch.py`

Suggested outputs:

- Human-readable edit report: Markdown/CSV
- Machine patch JSON
- Optional patched graph snapshot in a new folder

Do not overwrite `v2_curriculum_focus`; write a new snapshot such as:

- `graph\snapshots\v2_curriculum_focus_expert_review_YYYYMMDD`

### 6. Improve Review UX

Useful next UI features:

- Show edited nodes with a visible small badge.
- Add a filter for changed nodes/links only.
- Add a "review changes" panel listing all edits.
- Add comments/notes per edit.
- Add reason codes:
  - duplicate
  - wrong label
  - wrong relation
  - missing relation
  - out of scope
- Add confidence/acceptance status for expert edits.

### 7. Multi-Expert And Conflict Handling

If more than one expert will edit:

- Keep one session per expert.
- Do not merge in-browser.
- Merge offline by comparing patch JSONs.
- Treat node deletion and rename conflicts as review items.
- Treat added links as additive unless endpoints were deleted by another reviewer.

## Known Caveats

- The generated HTML embeds the graph, so each graph update requires rerunning:

```powershell
python curriculum_graph_et\scripts\build_v2_graph_viewer.py
```

- Browser `localStorage` is per browser/profile/path. Export JSON is the safest handoff format until backend save is wired.
- The current backend allows CORS from anywhere for local prototyping. Tighten this if it is ever exposed beyond localhost.
- The viewer currently does not authenticate experts. Add this before using it as shared infrastructure.
